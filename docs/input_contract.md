# Input Contract

ShortHaul Dispatch Agent accepts one scheduling scenario and one set of optimization settings. The recommended external input is a single Excel workbook. JSON and CSV remain available for advanced integrations.

## Recommended Excel Workbook

Download the template:

```http
GET /templates/workbook.xlsx
```

Preview it in a browser:

```http
GET /templates/view
```

Required sheets:

| Sheet | Purpose |
| --- | --- |
| `fleets` | Owned fleet resources |
| `routes` | Lanes, waves, latest dispatch times, travel times, fleet mapping, and costs |
| `demand` | Forecast volume by route |

Minimum required columns:

| Sheet | Required columns |
| --- | --- |
| `fleets` | `fleet_id`, `owned_vehicles` |
| `routes` | `route_id`, `origin`, `destination`, `wave`, `latest_dispatch_time`, `travel_min`, `fleet_id` |
| `demand` | `route_id`, `volume` |

`demand.ready_time` is optional. If it is blank, the adapter uses `latest_dispatch_time - 120 minutes` for that route.

Optional sheets:

| Sheet | Purpose |
| --- | --- |
| `compatibility` | Destination pairs that can be milk-run together |
| `settings` | Capacity, strategy, solver, and objective-weight overrides |

Upload the workbook through the UI or:

```http
POST /schedule/upload
Content-Type: multipart/form-data
```

Use multipart field name `workbook`.

To convert the workbook into the internal JSON payload locally:

```powershell
$env:PYTHONPATH="src"
python -m shorthaul_agent.cli build-payload --workbook examples/workbook_template/shorthaul_dispatch_template.xlsx --request examples/external_request.txt --output outputs/schedule_payload.json
```

## REST Payload

```json
{
  "request": "Schedule the short-haul plan. Minimize cost and improve owned-vehicle turnover.",
  "prefer_cpsat": true,
  "config_overrides": {
    "vehicle_capacity": 1000,
    "container_capacity": 800,
    "max_stops": 3,
    "allow_container": true,
    "allow_external": true,
    "tail_cover_strategy": "cost_aware",
    "objective_weights": {
      "cost": 1.0,
      "turnover": 0.5,
      "fill_rate": 0.2
    }
  },
  "instance": {
    "id": "external-instance",
    "date": "2024-12-16",
    "fleets": [],
    "routes": [],
    "forecast": []
  }
}
```

Submit it to:

```http
POST /schedule
```

Use `POST /validate-instance` before solving if the payload is generated by an external system.

The same JSON payload can also be uploaded as a local file through the Web UI or:

```http
POST /schedule/upload
Content-Type: multipart/form-data
```

Use the multipart field name `payload_json` for the complete JSON file.

## CSV Input

CSV is an advanced integration path. For manual use and first-time onboarding, use the Excel workbook above.

Required files:

- `fleets.csv`
- `routes.csv`
- `forecast.csv`

Optional files:

- `milk_run_pairs.csv`
- `config_overrides.json`

Templates are stored in [examples/csv_template](../examples/csv_template) and are also available through:

```http
GET /templates
```

These files can be selected directly in the Web UI upload section, or posted to:

```http
POST /schedule/upload
Content-Type: multipart/form-data
```

Expected multipart field names:

| Field | File |
| --- | --- |
| `fleets` | `fleets.csv` |
| `routes` | `routes.csv` |
| `forecast` | `forecast.csv` |
| `milk_run_pairs` | `milk_run_pairs.csv` |
| `config_overrides` | `config_overrides.json` |

The machine-readable schema is available through:

```http
GET /schema
```

## CSV Columns

`fleets.csv`

| Column | Required | Description |
| --- | --- | --- |
| `id` | yes | Fleet identifier used by routes |
| `vehicle_count` | yes | Number of owned vehicles |
| `fixed_cost` | no | Fixed cost in the planning horizon |
| `variable_cost_per_trip` | no | Owned-vehicle trip cost |
| `normal_load_minutes` | no | Loading time without container |
| `normal_unload_minutes` | no | Unloading time without container |
| `container_load_minutes` | no | Loading time with container |
| `container_unload_minutes` | no | Unloading time with container |

`routes.csv`

| Column | Required | Description |
| --- | --- | --- |
| `id` | yes | Unique route id |
| `origin` | yes | Origin site |
| `destination` | yes | Destination station |
| `wave` | yes | Dispatch wave label |
| `latest_dispatch_minute` | yes | Minute offset from planning start |
| `travel_minutes` | yes | Travel duration |
| `fleet_id` | yes | Fleet that can serve the route |
| `variable_cost` | no | Route-level owned-trip cost |
| `external_cost` | no | Explicit external-carrier cost |
| `external_cost_multiplier` | no | Multiplier when `external_cost` is omitted |

`forecast.csv`

| Column | Required | Description |
| --- | --- | --- |
| `route_id` | yes | Route id from `routes.csv` |
| `minute` | yes | Demand time bucket as minute offset |
| `volume` | yes | Forecast volume |

`milk_run_pairs.csv`

| Column | Required | Description |
| --- | --- | --- |
| `left_destination` | yes | First compatible destination |
| `right_destination` | yes | Second compatible destination |

## Time Representation

CSV templates use minute offsets from the planning start to avoid ambiguity across day boundaries. For example:

- `1380` means D+0 23:00.
- `1800` means D+1 06:00.
- `2280` means D+1 14:00.

JSON also accepts `HH:MM` and `HHMM` for same-day times, but minute offsets are recommended for operational integrations.

## Build A Payload From CSV

```powershell
$env:PYTHONPATH="src"
python -m shorthaul_agent.cli build-payload --csv-dir examples/csv_template --request examples/external_request.txt --output outputs/schedule_payload.json
```

Then call:

```http
POST /schedule
```

with the generated `outputs/schedule_payload.json`.

## Server-Local CSV Solve

For local deployments where the API server can access the CSV folder directly:

```http
POST /schedule/from-csv-dir
```

```json
{
  "data_dir": "examples/csv_template",
  "request": "Schedule the external short-haul scenario.",
  "instance_id": "external-instance",
  "date": "2024-12-16",
  "prefer_cpsat": true,
  "config_overrides": {}
}
```
